五：Transform组件。

1.Position、Rotation和Scale。

（1）Position。

注：子物体面板上的坐标，默认是localPosition(相对于父类的坐标)，所以如果要让子物体的坐标要和世界坐标位置一致，那么需要改变的是transform.position。

（2）Rotate方法和Rotation属性。

注：

1.Rotate()方法是：旋转多少度。在原有的基础上累加，即旋转了多少角度。又旋转了多少角度，是在原有的基础上在旋转。

2.rotation属性是：旋转到某个角度，就是是在update中每帧都执行（我这里测试是放在了update中）。但每次旋转到的角度动是5，所以一直都是旋转到5度。

3.比如你只想让他旋转到多少,用rotation;假如想让他一直转,可以用Rotate。

（3）eulerAngles属性。

注：

1.Unity面板上Inspector中Rotation的值对应的是transform.eulerAngles值，而不是transform.rotation值，但transform.eulerAngles这个变量仅用于读取和设置角度为绝对值。

2.如果要图形一直旋转，不要使用transform.eulerAngles递增，因为当超过角度360度，它将错误，要使用Transform.Rotate替代。

3.不要分别设置欧拉角其中一个轴（例如： eulerAngles.x = 10; ），因为这将导致偏移和不希望的旋转。当设置它们一个新的值时，要同时设置全部；

例如：（float x = 10；float y = 10；float z= 10 ；transform.eulerAngles = new Vector3(x, y, z);(10,10,10)）

八：Unity生命周期函数。

1.Awake()。

特点：

1.最早执行。

2.如果该物体未启用，那么启用后就会执行，后续无论再关闭启用都没用，只会执行一次。

2.OnEnable()。

特点：

1.如果该物体未启用，那么启用后就会执行，后续再关闭，然后启用就会执行，和Awake()不同，OnEnable()只要物体启用一次就会执行一次。

2.根据特性，适用于类似于打开背包的功能，每次打开游戏背包都会启用执行一次。

3.Start()。

特点：

1.和Awake()一样，只会执行一次，后续无论做什么都没用。

2.和Awake()唯一的差别在于，执行顺序在Awake()和OnEnable()之后。

4.FixedUpdate()。

特点：

1.只要运行程序就会不断执行。

2.和Update的区别在于：是以固定秒数不断执行的，默认是0.02秒执行一次。

5.Update()。

特点:

1.只要运行程序就会不断执行。(即便游戏暂停也在执行)

2.和FixedUpdate的区别在于：不是以固定时间执行的，是以每帧执行一次，因此每台电脑执行速率不同，帧率越高，Update能够执行越多。

3.可以根据查看Time.deltaTime来查看渲染上一帧的时间，因为各种因素的影响，时间是一直在变化的。

6.LateUpdate()。

特点：

1.和Update()一样，运行程序会不断执行，且是以帧来执行的。

2.不同点在于：执行顺序在Update()之后。

7.OnDisable()。

特点：

1.和OnEnable()相反，是在每次禁用物体时触发。

8.OnDestroy()。

特点：

1.当物体被销毁时执行。

2.销毁物体之前，物体先会被禁用，所以之前还会执行一次OnDisable()。

九：Invoke函数。(负责延迟方法的执行)

十：协程Coroutine(corporate routine)。

1.协程特点。

特点：

  1. 协程在中断指令(YieldInstruction)产生时暂停执行。

  2. 协程一暂停执行便立即返回 //中断协程后返回主函数，暂停结束后继续执行协程剩余的函数。

  3. 中断指令完成后从中断指令的下一行继续执行。

  4. 同一时刻、一个脚本实例中可以有多个暂停的协程，但只有一个运行着的协程。

  5. 函数体全部执行完后，协程结束。

  6. 协程可以很好的控制跨越一定帧数后执行的行为。

  7. 协程在性能上、相比于一般函数几乎没有更多的开销。

2.StartCoroutine开启一个协程(有两种主要方式)。

（1）StartCoroutine(IEnumerator routine)。

StartCoroutine(Example());（注意方法名后加括号，参数可写在括号里）

优点：灵活，性能开销小。

缺点：无法单独的停止这个协程，如果需要停止这个协程只能等待协同程序运行完毕或则使用StopAllCoroutine();方法。

（2）StartCoroutine (methodName:string, value : object = null)。

StartCoroutine(“string methodName”,value);(注意双引号，value为想传递的参数)

优点：可以直接通过传入协同程序的方法名来停止这个协程：StopCoroutine(“string methodName”);(注意双引号)

缺点：性能的开销较大，只能传递一个参数。

3.停止协程。

  1. StopCoroutine(string methodName); //用于开启协程的第二种的停止
  2. StopAllCoroutine(); //用于开启协程的第一种的停止
  3. 设置gameobject的active为false时可以终止协同程序，但是再次设置为true后协程不会再启动。

4.yield return暂停协程，然后恢复执行。(yield return保存函数状态，迭代器块返回调用迭代的地方。等待下次调用继续从当前位置处理)

注：使用yield return SomeCortoutineMethod等同于表格的yield return StartCoroutine(“SomeCortoutineMethod”)。

5.协程的执行顺序。

开始协同程序 -> 执行协同程序 -> 中断协同程序（中断指令）-> 返回上层继续执行

->中断指令结束后继续执行协同程序剩下的内容

（1）例1：

  1. 先执行print(“Starting ”+Time.time);

  2. 执行StartCoroutine，开启协程，转到协程函数中第18行;

  3. 一直到21行遇到yield return，发现是暂停协程函数，所以跳出协程函数回到12行继续执行，2s之后再返回21行继续执行;

  4. 然后到14行立刻执行print(“Before WaitAndPrint Finished”)，因为是立刻返回执行，所以时间为0;

  5. 14行以后没东西执行，2s后返回21行，继续往下执行;

  6. 最后执行23行，因为前面有2s关于协程的暂停返回执行，所以后续输出也是2s左右。

（2）例2：

  1. 先执行第9行，开启协程，转到14行的协程函数中;

  2. 然后执行16行的yield return，发现是暂停协程函数，所以跳出协程函数回到9行继续执行，1s之后再返回16行继续执行;

  3. 返回第9行后，执行第十行的print(“开启协程了”);

  4. 经过1s后，返回第16行，然后执行19行的print(“Starting”+Time.time);

  5. 然后执行21行的yield return，发现是暂停协程函数并开始新的协程，直到新的协程全部执行完成才返回，所以跳到27行新的协程函数中;

  6. 在27行以后，到30行遇到yield return是暂停2s，因为前面是在yield return开始新的协程进入的，所以不会返回21行，而是等待2s后继续执行32行的print(“WaitAndPrint”);

  7. 新的协程函数全部执行完成，返回21行后继续执行;

  8. 执行23行的print(“Done”)。

（3）例3：如果StartCoroutine转到的协程函数中后续执行的是yield return新的协程，则会在遇到第二个yield return后返回第一个StartCoroutine。

（4）例4：结合了例题3和停止协程的知识，第二个yield return时会返回到第一个StartCoroutine执行。

6.协程的注意事项。

  1. 不能在Update或者FixUpdate方法中使用协同程序，否则会报错。

  2. 关于中断指令：

中断指令/YieldInstruction,一个协程收到中断指令后暂停执行，返回上层执行同时等待这个指令达成后继续执行。

2.时间类。(Time.deltaTime和Time.timeScale是重点)

Time.deltaTime：

  1. Time.delta表示的是上一帧的时间，比如你的帧率是2000，则它对应的时间就是1/2000。

  2. 因为Update是一帧执行一次，所以通常在游戏移动中，移动会乘Time.delta来造成速度的一致性。

  3. 因为你如果帧率为300，那么相同时间Update假设执行300次，则你的Time.delta就是1/300，相乘就是1;而另一人帧率为2000,那么相同时间Update执行2000次，而他的Time.delta为1/2000，所以相乘也是1。

  4. 帧率越高的设备，虽然移动是相同的，但是因为相同时间内帧数多，所以画面会更流畅。

（3）碰撞信息。

​ ——碰撞触发重要知识——

  1. 双方都没有碰撞体和刚体，是绝对不可能触发碰撞事件。(函数)
  2. 双方都有碰撞体和刚体，是可以触发的。
  3. 一方有刚体和碰撞体，另外一方只有碰撞体，是可以触发的。(无论哪一方都有，都可以进入碰撞事件)
  4. 双方都没有刚体，只有碰撞体，无法触发碰撞事件。

5. 2D物理系统：触发。

（1）设置触发，注意设置触发后该物体的碰撞体的物理边缘就消失了。

（2）触发事件。(和碰撞类似，不同点在于设置触发后物理边缘消失，就可以穿过任何物体)

​ ——触发的重要知识——

  1. 双方都没有碰撞体和刚体，是绝对不可能引起触发事件。
  2. 双方都有碰撞体和刚体，是可以引起触发事件的。
  3. 一方有刚体和碰撞体，另外一方只有碰撞体，是可以引起触发事件的。(无论哪一方都有，两者都可以进入触发事件)
  4. 双方都没有刚体，只有碰撞体，无法引起触发事件。

（3）如何实现一直在触发中。(例如实现踩地刺然后一直掉血)

  1. 步骤有3个元素，地面(碰撞体)、地刺(打上IsTrigger的碰撞体,没刚体保证不下落)、玩家(刚体和碰撞体);

  2. 玩家因为刚体的重力下落，接触到地刺上，但地刺因为物理边缘消失直接穿过，最后落在加了碰撞体的地面上;

  3. 因为地刺的触发范围和地面贴近，所以玩家就会在其中一直执行触发事件。

  4. 另外，因为刚体休眠模式的问题，物体静止后因为Start Awake而不执行触发中了，所以我们可以修改其休眠模式为Never Sleep。

  5. 然后在OnTriggerStay2D(Collider2D collision)中修改对应的代码即可。
